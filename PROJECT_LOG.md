# 青柠集项目说明

## 项目名称

青柠集 - 校园闲置交换 App 原型

## 项目定位

青柠集是一个面向校园场景的本地可运行闲置交换 App 原型。项目围绕学生在校内转让教材、数码、宿舍用品、设计工具等物品的需求，提供从浏览、筛选、发布、管理到联系卖家的完整基础流程。

当前项目用于课程展示、作品集说明和功能原型验证，暂未上线，也没有接入真实用户系统或线上数据库。

## 目标用户

- 有闲置物品想在校内转让的学生
- 想低价购买教材、工具、宿舍用品的学生
- 需要快速了解商品状态和交易地点的校园用户
- 产品设计、交互设计课程中需要展示完整 App 流程的项目使用者

## 核心问题

校园闲置交易常见问题是信息分散、商品状态不清楚、沟通入口不统一、交易地点不明确。青柠集希望用一个轻量 App 原型，把发布、浏览、收藏、状态管理和联系卖家集中到同一个流程里，让校园闲置物品更容易被发现和流转。

## 前端结构

- `frontend/index.html`：电脑端展示页，用于浏览项目整体视觉和基础入口。
- `frontend/mobile.html`：手机端 App 原型主文件，包含首页、分类、发布、商品详情、我的、消息会话、聊天详情等主要交互。

手机端当前采用单 HTML 文件实现，页面内包含结构、样式和脚本，方便本地演示和快速迭代。

## 后端结构

- `backend/server.js`：Node.js 后端入口，使用原生 `http` 模块提供 API。
- `backend/data/products.json`：当前商品数据。
- `backend/data/demoProducts.json`：演示商品数据，用于重置。
- `backend/data/transactions.json`：商品状态变化产生的交易记录。
- `backend/package.json`：后端运行脚本。
- `backend/README.md`：后端接口与运行说明。

## 主要接口列表

- `GET /health`：后端健康检查。
- `GET /api/categories`：获取商品分类。
- `GET /api/products`：获取商品列表。
- `GET /api/products/:id`：获取单个商品详情。
- `POST /api/products`：新增商品。
- `PATCH /api/products/:id`：编辑商品基础信息。
- `PATCH /api/products/:id/status`：修改商品状态。
- `DELETE /api/products/:id`：删除商品。
- `GET /api/transactions`：获取交易记录。
- `POST /api/reset-demo-products`：恢复演示商品数据，并清空交易记录。

## 当前技术栈

- 前端：HTML、CSS、原生 JavaScript
- 后端：Node.js 原生 `http` 模块
- 数据：本地 JSON 文件
- 图片处理：前端 Canvas 压缩后转为 Base64 数据
- 本地状态：`localStorage` 用于收藏和消息会话保留
- 版本管理：Git

## 当前项目状态

当前项目是一个本地可运行的校园闲置交换 App 原型，已经具备较完整的核心用户流程。它适合用于作品集展示、课程汇报和继续做交互细化。

项目仍处于原型阶段，没有真实登录、权限系统、线上部署、真实支付或实时聊天能力。后端数据使用本地 JSON 文件模拟，消息会话主要保存在浏览器本地。

## 2026-05-26 收藏数据迁移节点

- 收藏数据已从前端 `localStorage` 迁移到 Supabase `favorites` 表。
- 收藏关系与当前登录用户的 `user_id` 绑定，并保存 `user_email`。
- 未登录用户不能收藏商品，前端提示“请先登录后收藏”。
- “我的收藏”改为读取当前用户的后端收藏列表，不再以 `localStorage` 作为主要数据源。
- 旧版 `localStorage` 收藏数据仅保留兼容读取，不再写入新的收藏状态。

## 2026-05-27 消息会话迁移节点

- 消息会话已迁移到 Supabase `conversations` / `messages` 表。
- “联系卖家”会为当前登录买家和商品创建或复用真实会话，同一个买家对同一个商品不会重复创建。
- 会话列表读取当前登录用户作为买家或卖家参与的 `conversations`，并按 `last_message_at` 倒序展示。
- 聊天详情读取 `messages` 表，发送消息会写入 `messages` 并同步更新 `conversations.last_message` / `last_message_at`。
- 会话与 Supabase Auth 登录用户绑定；未登录用户不能创建、读取或发送消息。
- 非会话参与者读取聊天记录或发送消息会返回 403。

## 2026-05-27 用户资料与卖家身份绑定节点

- Supabase 新增 `profiles` 表，用于保存当前登录用户的邮箱、昵称、学校和专业。
- 后端新增 `GET /api/profile` 和 `PATCH /api/profile`，用于读取和更新当前登录用户资料。
- 用户没有 profile 时，后端会按 Supabase Auth 用户自动创建默认 profile，昵称默认使用邮箱前缀。
- 商品发布和编辑时，`seller` 不再完全信任前端传值，而是绑定当前登录用户 profile 的 `display_name`。
- 如果 profile 昵称为空，商品 `seller` 会回退为当前登录邮箱前缀。

## 2026-05-27 订单系统 v1 节点

- Supabase 新增 `orders` 表，承载买家下单、模拟支付、自取完成和取消流程。
- 后端新增订单接口：`POST /api/orders`、`GET /api/orders`、`PATCH /api/orders/:id/pay-simulate`、`PATCH /api/orders/:id/complete`、`PATCH /api/orders/:id/cancel`。
- 当前订单支付为模拟支付，不接真实微信、支付宝，也不做青柠币。
- 订单状态包括：`待支付`、`待自取`、`已完成`、`已取消`。
- 模拟支付成功后，订单进入待自取，并把商品状态改为 `已预定`。
- 订单完成后，商品状态改为 `已出`，并写入一条 `transactions` 交易记录。
- 后续可在该流程基础上接入支付宝或微信支付沙箱。

## 2026-05-27 订单系统 v1.1 收口节点

- 创建订单前会检查同一商品是否已有 `待支付` 或 `待自取` 的进行中订单，避免重复下单。
- 模拟支付前会再次确认商品仍是 `在售`，商品状态变化时返回“商品状态已变化，无法支付”。
- 完成订单要求订单处于 `待自取` 且 `payment_status` 为 `已支付`，已完成或已取消订单不能重复操作。
- 取消进行中订单后，如果商品当前仍是 `已预定`，会释放回 `在售`。
- 当前仍为模拟支付流程，不接真实微信、支付宝，也不做青柠币。
